# 59단계 최종 운영 전 사용자/관리자 QA 시나리오 및 잔여 리스크 점검

## 범위

- 대상: 구매+교체서비스, 대여+교체서비스, 일반 교체서비스, 일반 상품 주문, 일반 대여 주문, 관리자 주문/교체서비스/대여 화면.
- 원칙: 결제 구조, 재고 차감, DB schema, 상태 enum, 리뷰 eligibility, ServiceReviewCTA, 도메인 구조, 관리자 대시보드 구조는 변경하지 않음.
- 결과: 코드 변경 없이 운영 전 QA 체크리스트와 코드 기준 리스크 점검 결과만 문서화.

## 실제 확인한 사용자 시나리오

1. 스트링 구매 + 교체서비스: `/products/[stringId]` → 교체서비스 포함 신청 → 체크아웃 → `/mypage?tab=orders` → 주문 상세 → 교체서비스 상세.
2. 라켓 구매 + 스트링 선택 + 교체서비스: `/rackets/[racketId]` → 구매하기 → 스트링 선택 → 체크아웃 → 주문 상세 → 교체서비스 상세.
3. 대여 + 교체서비스: 대여 상품 → 대여+교체서비스 신청 → 결제 → 대여 상세 → 장착 정보 → 출고/반납/수령확인.
4. 일반 standalone 교체서비스: `/services/apply` → 신청서 작성 → 교체서비스 상세 → 운송장 또는 방문/수거 → 교체서비스 확정.
5. 일반 상품 주문: 상품 주문 → 주문 상세 → 배송 완료 → 구매확정 → 리뷰 작성.
6. 일반 대여 주문: 대여 주문 → 대여 상세 → 출고 확인 → 반납 운송장 등록 → 반납 완료 → 수령확인.

## 실제 확인한 관리자 시나리오

1. 관리자 주문: `/admin/orders`, `/admin/orders/[orderId]`에서 일반 주문, 교체서비스 포함 주문, 취소/환불/구매확정 주문의 상세 표시 경로 확인.
2. 관리자 교체서비스: `/admin/applications`, 신청서 상세 컴포넌트에서 standalone/order/rental 연결 구분과 상태 변경 제한 확인.
3. 관리자 대여: `/admin/rentals`, `/admin/rentals/[rentalId]`에서 일반 대여, 대여+교체서비스, 출고/반납/수령확인 상태 확인.

## 금지 키워드 재검색 결과 분류

### A. 사용자/관리자 UI 노출이므로 수정 필요

- 없음.

### B. 일반 standalone 교체서비스라 유지

- `교체서비스 상세`, `교체서비스 상세 보기`: 단독 신청서 또는 보조 장착 정보 진입 CTA로 유지.
- `단독 교체서비스는 교체서비스 상세에서 확정할 수 있습니다.`: standalone 확정 흐름 안내로 유지.

### C. 코드 변수명/상태값/DB 필드라 유지

- 없음.

### D. 테스트/주석/문서라 유지

- `결제 대기`, `결제 대기(24h+)`: 관리자 대시보드/운영 지표 및 주석에서 결제 대기 상태를 설명하는 용도라 유지.
- `OrderList`의 과거 주석/분기: 현재 `/mypage?tab=orders` 대표 목록은 `TransactionFlowList`가 담당하며, legacy 탭 컴포넌트의 문구로 분류.

### E. 후속 과제

- `OrderList` legacy 컴포넌트가 다시 라우팅에 연결될 경우, 연결 주문의 대표 CTA가 주문 상세 중심 정책과 충돌하지 않는지 재확인 필요.

## 최종 위험 체크

1. 주문 기반 신청서에 standalone 결제 CTA가 뜨는 조건: 연결 신청서에서는 주문 상세 중심 href와 상태 안내가 우선되어 치명 오노출 없음.
2. 대여 기반 신청서에 입고/자가발송 CTA가 뜨는 조건: 대여 연결 여부를 기준으로 대여 상세 중심 이동과 보조 장착 정보 확인만 노출되는 구조로 확인.
3. 취소된 주문/신청서/대여에 해야 할 일 CTA가 남는 조건: 취소 상태 분기와 상태별 CTA 제한이 존재함.
4. 구매확정과 교체서비스 확정이 동시에 뜨는 조건: 주문 연결 신청서는 주문 구매확정과 함께 처리된다는 안내로 분리됨.
5. 수령확인과 교체서비스 확정이 동시에 뜨는 조건: 대여 연결 신청서는 대여 수령확인과 함께 처리된다는 안내로 분리됨.
6. 리뷰 CTA가 확정 전 노출되는 조건: 주문 리뷰 CTA는 `userConfirmedAt` 또는 `구매확정` 상태 기준으로 제한됨.
7. 스트링 단품 판매 버튼이 다시 노출되는 조건: 스트링 단품 주문 정책 플래그와 장착비 정책 분기가 있어 기본 단품 판매 노출 위험은 낮음.

## 시나리오별 QA 결과

### A. 스트링 구매 + 교체서비스

- 스트링 상세는 단품 판매 정책 플래그와 장착 가능 스트링 정책을 기준으로 분기한다.
- 마이페이지 통합 목록은 주문 연결 신청서의 대표 CTA를 주문 상세로 보낸다.
- 주문 상세 및 신청서 상세은 선택 스트링/게이지/색상/장착비/텐션/요청사항과 연결 주문 상품 확인 경로를 가진다.
- 주문 기반 신청서에서 standalone 결제 CTA, 라켓 발송/입고 운송장 CTA가 대표 액션으로 노출되는 치명 조건은 확인되지 않았다.

### B. 라켓 구매 + 스트링 선택 + 교체서비스

- 라켓 상세의 구매 경로와 별도 `select-string` 단계가 존재한다.
- 주문 상세 중심으로 라켓/선택 스트링/게이지/색상/장착비 확인이 가능하다.
- 별도 교체서비스 결제 CTA나 라켓 발송/입고 운송장 CTA가 주문 연결 건의 대표 액션으로 노출되는 치명 조건은 확인되지 않았다.

### C. 대여 + 교체서비스

- 통합 목록은 대여 연결 신청서의 대표 CTA를 대여 상세로 보낸다.
- 대여 상세는 장착 정보와 출고/반납 정보를 분리하여 보여준다.
- 금지 문구인 `이용확정` 계열은 검색 결과 없음.
- 대여 기반 신청서에서 입고/자가발송/라켓 발송 CTA가 대표 액션으로 노출되는 치명 조건은 확인되지 않았다.

### D. 일반 standalone 교체서비스

- orderId/rentalId가 없는 신청서는 교체서비스 상세 중심으로 유지된다.
- 고객 라켓 입고/자가발송/방문 수거 흐름과 standalone 확정 안내는 유지한다.
- standalone 전용 결제/운송장/확정 CTA는 연결 신청서 정책과 구분되어 유지한다.

### E. 일반 상품 주문

- 일반 상품 주문은 주문 상세 중심으로 이동한다.
- 교체서비스 연결 정보가 없는 주문에는 교체서비스 장착 요약/보조 CTA가 붙지 않는 구조다.
- 구매확정 및 리뷰 CTA는 배송 완료/구매확정 상태 기준으로 제한된다.

### F. 일반 대여 주문

- 일반 대여는 대여 상세 중심이다.
- 교체서비스 연결이 없는 경우 장착 정보 CTA가 조건 없이 붙는 구조는 확인되지 않았다.
- 수령확인 문구가 사용되며 `이용확정` 잔존 검색 결과는 없다.

## 관리자 QA 결과

- 관리자 주문 상세는 교체서비스 연결 주문에서 선택 스트링/게이지/색상/장착비/텐션/요청사항을 확인하는 흐름을 가진다.
- 관리자 교체서비스 상세는 orderId/rentalId 연결 여부에 따라 상태 변경 위치와 연결 상세 이동을 구분한다.
- 관리자 대여 상세는 대여 결제/출고/반납 영역과 교체서비스 장착/작업 상태 영역을 분리한다.

## 결론

- 이번 단계에서 코드 수정은 하지 않았다.
- 결제, 재고, DB schema, 상태 enum, 리뷰 eligibility, ServiceReviewCTA, 도메인 구조 변경 없음.
- 배포 전 수동 QA에서는 실제 결제 sandbox 데이터로 각 시나리오의 금액 합산, 연결 ID 생성, 상태 전이를 최종 확인해야 한다.
